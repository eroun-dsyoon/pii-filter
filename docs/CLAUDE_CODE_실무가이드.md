## 1. 전체 프로세스 요약

```
요구사항 문서 업로드
    ↓
초기 프롬프트 (전체 서비스 설계 + 구현 요청)
    ↓
1차 구현 → 테스트 실행 → 오류 수정
    ↓
합성 데이터 10,000건 생성
    ↓
Blue Team 평가 (v1: F1=0.9649, FP=321)
    ↓
Haiku Agent(Judge)로 오탐/미탐 분석
    ↓
피드백 반영 → 엔진 개선
    ↓
재평가 (v3: F1=0.9862, FP=0) ← 3회 반복
    ↓
기능 추가 (은행 식별, 관리자 페이지 개선)
    ↓
배포
```

**총 소요 시간**: 약 1시간
**반복 횟수**: 엔진 개선 3회, 버그 수정 5회
**최종 성과**: Precision 100%, F1 98.62%

## 2. Phase 1: 요구사항 문서 준비

### 핵심 원칙: "Claude에게 줄 문서는 상세할수록 좋다"

우리가 제공한 요구사항 문서에는 다음이 포함되어 있었습니다:

| 항목 | 내용 | 왜 중요한가 |
|------|------|------------|
| Entity Type 목록 | RRN, CRN, Phone 등 8가지 | Claude가 구현 범위를 정확히 파악 |
| 숫자 대체 문자 매핑표 | 이모지, 한글, 한자, Leet 등 전체 매핑 | 정규식 패턴과 char_map 구현의 기반 |
| 3단계 검출 레벨 정의 | Level 1/2/3 각각의 검출 범위 | 엔진 아키텍처 설계의 기반 |
| 검출/비검출 예시 | ✅ 검출 예시 + ❌ 제외 예시 | 테스트 케이스와 오탐 방지 규칙의 기반 |
| API 응답 형식 | JSON 스키마와 normalized 필드 | API 설계를 정확하게 수행 |
| 오탐 시나리오 | "lollollolloll" 등 | 오탐 방지 로직의 출발점 |

### 문서 작성 팁

```markdown
✅ 좋은 예: 구체적인 입출력 예시 포함
"010 - 1234 _ 5678" → 검출 (공백+하이픈+언더스코어 혼용)
"0-1-0-1-2-3-4-5-6-7-8" → 제외 (각 숫자 사이 구분자)

❌ 나쁜 예: 추상적 설명만
"다양한 구분자가 포함된 전화번호를 검출해야 합니다"
```

> **교훈**: 요구사항 문서의 품질이 1차 구현의 정확도를 결정합니다.
> 예시가 많을수록 Claude가 엣지 케이스를 더 잘 처리합니다.

## 3. Phase 2: 초기 프롬프트 설계

### 실제 사용한 첫 프롬프트

```
업로드 한 요구사항 문서에 최적화된 서비스를 개발하고 git에 서비스를 오픈하고 싶습니다.

[최종 서비스]
- 사용자가 문장/파일을 입력하면 개인정보 존재유무, 어떤 개인정보가 존재하는지 알려주는 서비스
- 오탐이 발생했을 때, 오탐을 신고하는 매뉴
- 필터링 속도 : 1건당 0.01초 이내

[모델 개발 단계]
- 요구사항 문서를 상세하게 분석하여 개인정보를 단계별로 필터링하는 모델 개발
- 에이전트:
  1. 단계별로 설명한 개인정보를 합성데이터로 생성하여 필터링을 피하려고 노력하는 레드팀 에이전트
  2. 각 문장에 개인정보가 존재하는지 판단하는 블루팀 에이전트
  3. 블루팀 에이전트가 오탐을 했는지, 미탐을 했는지 정확히 판정을 해주는 판사 에이전트
- 합성 데이터는 총 10만개를 생성하여 기본 모델을 만든다.
- 관리자 페이지에서는 합성데이터를 몇개 생성할지 입력하게 한다.
- 생성된 합성데이터는 별도의 파일로 저장해 둔다.
- 알고리즘은 개선될때마다 필터링 수치를 지속적으로 보고서 형태로 만든다.
```

### 이 프롬프트가 효과적이었던 이유

| 요소 | 설명 |
|------|------|
| **최종 형태 명시** | "서비스", "관리자 페이지" → FastAPI + 웹 UI로 결정 |
| **비기능 요구사항** | "0.01초 이내" → 정규식 기반 설계 (LLM 호출 X) |
| **에이전트 역할 분리** | Red/Blue/Judge 3가지 → 명확한 모듈 분리 |
| **산출물 명시** | "합성데이터 파일", "보고서" → 출력 형식이 구체적 |
| **요구사항 문서 첨부** | 상세 스펙이 이미 정의됨 → 해석의 여지 최소화 |

### 프롬프트 구조 패턴

```
1. 컨텍스트 제공 (요구사항 문서 첨부)
2. 최종 목표 명시 (어떤 서비스를 원하는지)
3. 제약 조건 (속도, 형식 등)
4. 아키텍처 힌트 (에이전트 구조)
5. 산출물 정의 (데이터 파일, 보고서)
```

## 4. Phase 3: 에이전트 아키텍처 설계

### 3개 에이전트의 역할과 구현 방식

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  🔴 Red Team │ ──→ │ 🔵 Blue Team │ ──→ │  ⚖️ Judge   │
│  합성데이터    │     │  필터링 엔진  │     │  오탐/미탐   │
│  생성기       │     │  (정규식)     │     │  판정       │
└─────────────┘     └─────────────┘     └─────────────┘
                           ↑                     │
                           └─────── 피드백 ────────┘
```

### 설계 시 내린 핵심 결정들

#### 결정 1: Red Team을 알고리즘 기반으로 전환

```
초기 계획: Haiku API로 합성 데이터 생성
   ↓ 문제 발생: API 키 필요, 비용 발생, 속도 느림
   ↓
최종 결정: 알고리즘 기반 생성 + Haiku Agent로 창의적 샘플 보완
```

**교훈**: LLM이 항상 최선은 아닙니다.
- 규칙이 명확한 데이터 생성 → **알고리즘** (빠르고, 무료, 결정적)
- 창의적 우회 시도 생성 → **LLM** (다양하고, 예측 불가능)

#### 결정 2: Blue Team을 정규식 기반으로 구현

```
속도 요구사항: 1건당 0.01초(10ms) 이내
   ↓
LLM 기반 필터링: ~500ms/건 → ❌ 불가
정규식 기반 필터링: ~0.15ms/건 → ✅ 67배 여유
```

#### 결정 3: Judge를 Claude Code의 Haiku Agent로 실행

```
초기: Anthropic API 직접 호출 (API 키 필요)
   ↓ 사용자 피드백: "Claude Code 환경에서 Haiku를 사용하라"
   ↓
최종: Agent(model="haiku")로 오류 분석 → 로컬 패턴 분석으로 보완
```

**프롬프트 (Judge Agent에게 준 실제 프롬프트)**:

```
당신은 한국 개인정보(PII) 필터링 시스템의 Judge(판사) 에이전트입니다.

## 오탐(False Positive) 분석 - 총 321건
주요 패턴:
1. "2024-0312-4567" (주문번호) → BANK_ACCOUNT, PHONE으로 오탐
2. "978-89-6848-123-4" (ISBN) → BANK_ACCOUNT으로 오탐
...

## 미탐(False Negative) 분석 - 총 176건
주요 패턴:
1. 여권번호에 대체 문자가 포함된 경우: "Gⅸ🔟666⑻6"
...

다음 JSON 형식으로 응답하세요:
{
  "fp_analysis": { "root_cause": "...", "rule_improvements": [...] },
  "fn_analysis": { "root_cause": "...", "pattern_improvements": [...] },
  "priority_fixes": [...]
}
```

> **교훈**: Judge에게 줄 프롬프트에는 **실제 오류 데이터**를 포함해야 합니다.
> "오류를 분석해줘"보다 "이 321건의 오탐 패턴을 분석해줘"가 100배 효과적입니다.

## 5. Phase 4: 합성 데이터 생성

### 데이터 구성

```python
# 총 10,000건
# PII 포함: 7,000건 (70%) - 8가지 유형 × 3 레벨 우회 시도
# PII 미포함: 3,000건 (30%) - 숫자가 포함되지만 개인정보가 아닌 텍스트
```

### PII 포함 데이터 (Positive 샘플)

| 레벨 | 예시 | 비율 |
|------|------|------|
| Level 1 | `010-1234-5678` | ~33% |
| Level 2 | `010 - 1234 _ 5678` | ~33% |
| Level 3 | `공일공-일이삼사-오육칠팔` | ~33% |

### PII 미포함 데이터 (Negative 샘플) - 매우 중요!

```python
# 이것들이 오탐을 잡아내는 핵심 데이터
NON_PII_TEXTS = [
    "주문번호 2024-0312-4567",        # 계좌번호와 형식 유사
    "ISBN 978-89-6848-123-4",          # 숫자-하이픈 패턴
    "계약기간: 2024.01.01 ~ 2024.12.31", # 날짜
    "IP 주소: 192.168.0.1",           # 숫자-점 패턴
    "모델명 SM-G998N, 가격 1,350,000원", # 숫자 포함 일반 텍스트
    ...
]
```

> **교훈**: Negative 샘플의 품질이 Precision(정밀도)을 결정합니다.
> "개인정보처럼 보이지만 아닌 것"을 충분히 포함해야 합니다.

### 데이터 파일 형식

```jsonl
{"text": "전화번호는 010-1234-5678 입니다.", "has_pii": true, "pii_type": "PHONE", "pii_value": "010-1234-5678", "evasion_level": 1, "evasion_technique": null}
{"text": "주문번호 2024-0312-4567", "has_pii": false, "pii_type": null, "pii_value": null, "evasion_level": null, "evasion_technique": null}
```

## 6. Phase 5: 평가 → 분석 → 개선 반복

### 반복 과정 상세

#### 1차 평가 (v1)

```
10,000건 평가 결과:
  TP: 6824 | TN: 2679
  FP: 321  | FN: 176
  Precision: 0.9551 | Recall: 0.9749
  F1: 0.9649 | Accuracy: 0.9503
```

**문제 발견**:
- FP 321건 → 주문번호, ISBN, 날짜가 계좌번호로 오탐
- FN 176건 → 여권번호에 대체문자 포함 시 미탐

#### Judge Agent 분석 결과

```json
{
  "priority_fixes": [
    "1. BANK_ACCOUNT 정규식 범위 축소 (FP 감소 예상: ~200건)",
    "2. 날짜 형식 명시적 제외 (FP 감소 예상: ~80건)",
    "3. 여권번호 Level 3 처리 개선 (FN 감소 예상: ~150건)"
  ]
}
```

**이 분석을 받아 Claude에게 준 명령**:

```
Judge 분석 결과를 기반으로 필터링 엔진을 개선합니다.
```

→ Claude가 자동으로 코드 수정 (날짜 제외 규칙, 계좌번호 패턴 축소 등)

#### 2차 평가 (v2) - 과보정 발생

```
  FP: 93 (↓228) | FN: 669 (↑493)
  Precision: 0.9855 (↑) | Recall: 0.9044 (↓↓)
```

**문제**: 계좌번호 패턴을 너무 엄격하게 줄여서 정상 계좌도 미탐

**추가 명령**:
```
계좌번호 패턴을 다시 조정하고, 주문번호 오탐도 수정합니다.
```

#### 3차 평가 (v3) - 최적 균형 달성

```
  FP: 0 (↓93)   | FN: 190 (↓479)
  Precision: 1.0000 | Recall: 0.9729
  F1: 0.9862 | Accuracy: 0.9810
```

### 성능 변화 추이

```
버전  Precision  Recall   F1      FP    FN
─────────────────────────────────────────────
v1    95.51%    97.49%   96.49%  321   176
v2    98.55%    90.44%   94.32%   93   669  ← 과보정
v3   100.00%    97.29%   98.62%    0   190  ← 최적
```

### 각 반복에서 적용된 개선 사항

| 반복 | 개선 내용 | 영향 |
|------|----------|------|
| v1→v2 | 계좌번호 패턴을 은행별 실제 형식으로 제한 | FP ↓228, FN ↑493 |
| v1→v2 | 날짜/ISBN/문서번호/IP 제외 규칙 추가 | FP ↓ |
| v2→v3 | 계좌번호 넓은 패턴 복원 + validator 검증 | FN ↓479 |
| v2→v3 | 연도(2000~2099)로 시작하는 숫자열 제외 | FP ↓93 |
| v2→v3 | 전화번호 지역번호 범위 검증 강화 | FP ↓ |
| v2→v3 | 다른 타입 간 80% 이상 겹침 중복 제거 | FP ↓ |

> **교훈**: 한 방향으로 과보정하면 반대쪽이 무너집니다.
> Precision과 Recall의 균형을 항상 확인하세요.

## 7. Phase 6: 기능 추가 및 품질 개선

### 사용자 피드백 기반 개선

| 피드백 | 대응 | 프롬프트 |
|--------|------|---------|
| "합성데이터가 0건" | API 키 미설정 오류 → 에러 메시지 개선 + 알고리즘 기반 전환 | "API_KEY가 없으면 CLI를 통해서 생성하세요" |
| "관리자 화면에 개선 과정 정보가 없다" | FP/FN 샘플, before/after 비교, 개선 로그 추가 | "오탐/미탐 데이터에 대해 알고리즘을 개선하는 프로세스가 들어가 있나요?" |
| "계좌번호가 어디 은행인지 알려줘야" | 은행 식별 모듈 추가 | "계좌번호는 어디 계좌인지 알려줘야 사용자가 납득할 수 있을 것 같습니다" |
| "3단계에서 은행정보 안 보임" | normalized 값 기반 식별로 수정 | 스크린샷 첨부 |

> **교훈**: 스크린샷을 첨부하면 Claude가 문제를 즉시 파악합니다.
> "안 돼요"보다 "이 화면에서 이게 안 보여요"가 100배 효과적입니다.

## 8. 핵심 교훈 및 베스트 프랙티스

### 프롬프트 원칙

| # | 원칙 | 설명 |
|---|------|------|
| 1 | **요구사항 문서를 먼저 준비하라** | Claude에게 줄 문서가 상세할수록 1차 구현 품질이 높다 |
| 2 | **최종 형태를 명시하라** | "서비스", "API", "웹 UI" 등 산출물을 구체적으로 |
| 3 | **비기능 요구사항을 포함하라** | 속도, 보안, 형식 등 제약 조건이 아키텍처를 결정 |
| 4 | **한 번에 다 시키지 말고 단계적으로** | 구현 → 테스트 → 피드백 → 개선 사이클 |
| 5 | **오류 데이터를 직접 보여줘라** | "FP 321건의 패턴"처럼 구체적 데이터 제공 |
| 6 | **스크린샷을 적극 활용하라** | UI 문제는 말보다 스크린샷이 정확 |

### 데이터 원칙

| # | 원칙 | 설명 |
|---|------|------|
| 1 | **Positive와 Negative 비율은 7:3** | 과도한 Positive는 오탐을 놓침 |
| 2 | **Negative에 "헷갈리는 것"을 넣어라** | 주문번호, ISBN, IP 등 형식이 유사한 것 |
| 3 | **LLM보다 알고리즘 생성이 나을 때가 있다** | 규칙이 명확한 변형은 코드로 생성 |
| 4 | **10,000건이면 패턴 발견에 충분** | 더 많은 데이터보다 다양한 패턴이 중요 |

### 에이전트 설계 원칙

| # | 원칙 | 설명 |
|---|------|------|
| 1 | **역할을 명확히 분리하라** | 생성(Red), 실행(Blue), 평가(Judge) |
| 2 | **Judge의 출력 형식을 JSON으로 강제** | 구조화된 피드백이 자동 반영에 필수 |
| 3 | **피드백 루프를 반드시 포함하라** | Judge → Blue Team 피드백이 개선의 핵심 |
| 4 | **과보정을 감지하라** | Precision↑ + Recall↓이면 과보정 |

### 반복 개선 원칙

```
1회차: 전체 구현 + 기본 테스트 통과
2회차: 대량 데이터 평가 + 오류 패턴 발견
3회차: Judge 분석 + 개선 적용
4회차: 과보정 수정 + 균형 조정
5회차+: 사용자 피드백 반영 (UI, 기능)
```

### 템플릿 1: 초기 서비스 구현 요청

```markdown
[요구사항 문서 첨부]

위 요구사항 문서에 최적화된 서비스를 개발해주세요.

[최종 서비스]
- (서비스 형태: 웹, API, CLI 등)
- (핵심 기능 1~3개)
- (비기능 요구사항: 속도, 보안 등)

[아키텍처]
- (에이전트/모듈 구조)
- (각 모듈의 역할)

[산출물]
- (코드, 데이터 파일, 보고서 등)
```

### 템플릿 2: 합성 데이터 생성 요청

```markdown
다음 조건으로 합성 데이터를 생성해주세요:

- 총 개수: N건
- Positive 비율: 70% (PII 포함, 다양한 우회 기법)
- Negative 비율: 30% (PII 미포함, 형식이 유사한 비PII)
- 출력 형식: JSONL
- 필수 필드: text, has_pii, pii_type, evasion_level
```

### 템플릿 3: 평가 + 개선 요청

```markdown
합성 데이터 N건에 대한 평가 결과:
- FP: X건 (주요 패턴: ...)
- FN: Y건 (주요 패턴: ...)

이 오류 패턴을 분석하고 알고리즘을 개선해주세요.
개선 후 동일 데이터로 재평가하고 before/after를 비교해주세요.
```

### 템플릿 4: 사용자 피드백 기반 개선

```markdown
[스크린샷 첨부]

(구체적인 문제 설명)
예: "3단계로 검출했을 때 계좌번호에 은행 정보가 표시되지 않습니다"
```
